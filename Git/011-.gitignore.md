# 011 - .gitignore

## Introduction

Not every file inside a project should become part of the Git repository.

Many projects contain files that are specific to a developer's machine, generated automatically by tools, or contain sensitive information such as passwords or API keys.

These files do not belong to the project itself.

This is the purpose of **.gitignore**.

The important idea to remember is:

> **`.gitignore` defines which files should not become part of the repository.**

---

## Why Do We Need `.gitignore`?

Suppose our project contains the following files.

```text
Project/
│
├── main.c
├── README.md
└── passwords.txt
```

The file `passwords.txt` contains sensitive information.

If we execute

```bash
git add .
git commit -m "Initial commit"
```

Git will happily add every file.

Git does **not** inspect the contents of a file to determine whether it should be committed.

It simply follows the developer's instructions.

This behavior is intentional.

Git is designed to track files, not to decide which files are important or confidential.

---

## What Is `.gitignore`?

`.gitignore` is a normal text file that contains a list of rules.

Each rule tells Git which files or directories should be ignored when adding new files to the repository.

For example,

```text
passwords.txt
```

tells Git to ignore the file `passwords.txt`.

Likewise,

```text
*.log
```

ignores every file ending in `.log`.

Directories can also be ignored.

```text
build/
```

ignores the entire `build` directory.

---

## When Does Git Use `.gitignore`?

Git checks `.gitignore` when deciding whether a file should begin being tracked.

Suppose we execute

```bash
git add .
```

Conceptually, Git performs the following process.

```text
git add .
      │
      ▼
Find files
      │
      ▼
Does a file match a .gitignore rule?
      │
 ┌────┴────┐
 │         │
Yes       No
 │         │
 ▼         ▼
Ignore    Add to the staging area
```

Files that match an ignore rule are skipped.

They are not added to the staging area.

---

## Tracked vs Untracked Files

The most important limitation of `.gitignore` is that it only affects **untracked files**.

Suppose we accidentally commit

```bash
git add passwords.txt
git commit -m "Add passwords"
```

From this point onward, `passwords.txt` becomes a **tracked file**.

If we later create

```text
passwords.txt
```

inside `.gitignore`, Git **does not** stop tracking the file.

Future modifications are still detected because the file already belongs to the repository.

This is one of the most common misunderstandings when learning Git.

> **`.gitignore` prevents Git from tracking new files. It does not stop tracking files that already belong to the repository.**

---

## Stopping Git from Tracking a File

If a tracked file should no longer belong to the repository, we must remove it from Git's index.

The following command removes the file from the repository while keeping the local copy.

```bash
git rm --cached passwords.txt
```

Afterwards, we usually add the file to `.gitignore`

```text
passwords.txt
```

and create a commit.

From this point onward, the file remains on the developer's computer but is no longer tracked by Git.

Notice that this does **not** remove the file from previous commits.

If sensitive information has already been pushed to a remote repository, the secret should be considered compromised.

---

## Is `.gitignore` Part of the Repository?

Yes.

Although `.gitignore` controls local files, the `.gitignore` file itself is usually tracked.

This allows every developer working on the project to share the same ignore rules.

For example, if a project ignores

```text
.env
```

every collaborator automatically receives this rule after cloning the repository.

Each developer keeps their own local `.env` file, but none of these files become part of the repository.

---

## Summary

- `.gitignore` defines which files should not become part of the repository.
- Git does not analyze the contents of files before tracking them.
- `.gitignore` is consulted when Git decides whether to begin tracking new files.
- Files matching an ignore rule are skipped during `git add`.
- `.gitignore` only affects untracked files.
- Tracked files continue being tracked even if they later appear in `.gitignore`.
- `git rm --cached` removes a file from Git while keeping the local copy.
- `.gitignore` itself is usually tracked so the entire team shares the same ignore rules.
